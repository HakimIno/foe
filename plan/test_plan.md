# Master Test Plan & Failure Mitigation Guide (แผนการทดสอบและการป้องกันความล้มเหลวของโปรเจกต์)

เอกสารนี้ทำหน้าที่เป็นคู่มือการออกแบบเคสทดสอบ (Test Cases) การสกัดความผิดพลาดล่วงหน้า และขั้นตอนวิธีควบคุมเพื่อไม่ให้โปรเจกต์เว็บเบราว์เซอร์ล้มเหลว (Preventing Project Failure) จากปัญหาทางด้านสถาปัตยกรรม (เช่น Thread Deadlocks, GPU Memory Leaks, และ State Desynchronization)

---

## 🛑 3 ปัจจัยหลักที่ทำให้โปรเจกต์เบราว์เซอร์ล้มเหลว (และวิธีตั้งรับ)
1. **Thread Deadlock (UI ค้าง):** UI ของ Slint รันอยู่บนเธรดหลัก (Main Thread) แต่ Servo ทำงานแยกกันเป็นร้อยละสิบของเธรดย่อย หากเขียนระบบซิงค์ข้อมูล (Synchronous Lock เช่น `.lock().unwrap()`) ข้ามเธรดนี้โดยตรง UI จะหยุดการตอบสนองทันที
   - *วิธีตั้งรับ:* ใช้ **Asynchronous Message Passing** ผ่านช่องสัญญาณกู้อินฟอร์เมชัน (`tokio::sync::mpsc`) ห้ามจองล็อก mutex เป็นเวลานานในฟังก์ชันวาดผล UI
2. **GPU Memory Leak (แรมการ์ดจอหมด):** การแชร์ GPU Texture ระหว่าง Servo (OpenGL) และ Slint หากสลับแท็บไปมาแล้วไม่ทำลายหรือจัดเก็บ Texture เก่าที่หมดอายุลงอย่างเหมาะสม การ์ดจอจะแครชในเวลาไม่กี่นาที
   - *วิธีตั้งรับ:* ใช้ระบบ **Texture Pool / Cache** และผูกตัวทำลายคลาส (`Drop` trait) ใน Rust เพื่อสั่งรีลีสหน่วยความจำ GPU ทันทีที่ WebView ปิดตัวลง
3. **Rust Compiler Compatibility (คอมไพล์ไม่ผ่าน):** Servo พึ่งพา C++ bindings และมักต้องการ Rust compiler เวอร์ชันเจาะจง (มักเป็น Nightly) ซึ่งอาจขัดแย้งกับ Slint รุ่นล่าสุด
   - *วิธีตั้งรับ:* กำหนดเวอร์ชันของคอมไพเลอร์ที่ชัดเจนในไฟล์ `rust-toolchain.toml` ตั้งแต่วันแรก และใช้ Docker หรือการตั้งค่า CI/CD เพื่อคุมสภาพแวดล้อมการสร้างแอปพลิเคชัน

## 🔬 ตารางแผนการทดสอบแบบละเอียด (Phase-by-Phase Test Matrix)

### Phase 1: Project Setup & UI Foundation
*   **สิ่งที่ต้องทดสอบ (Test Cases):**
    - ความเข้ากันได้ของระบบจัดตำแหน่ง Layout (เมื่อเปิดแอปและทำการย่อ/ขยายหน้าต่างแบบฉับพลัน)
    - เวลาตอบสนอง (Event-Loop Latency) หลังจากกดปุ่มนำทาง (ต้องต่ำกว่า 16ms หรือ 60fps)
*   **วิธีการทดสอบ (Testing Method):**
    - `Unit Test`: เขียนแบบสอบตรวจโครงสร้างข้อมูล `TabInfo` และโมดูล SQLite
    - `Manual Test`: รันแอป สังเกตความยืดหยุ่นของแท็บแนวตั้ง (Arc Sidebar) และการแสดงผล Command Palette
*   **ความเสี่ยงและวิธีป้องกัน (Failure Mode & Mitigation):**
    - *ความเสี่ยง:* การวางเลย์เอาต์ซ้อนรูปวงกลมทำให้เกิด Binding Loop ใน Slint
    - *วิธีป้องกัน:* ป้องกันโดยการประกาศขนาดความกว้าง/ความสูงแบบระบุค่าคงที่ (Explicit Size) ให้กับกล่องคอนเทนเนอร์หลัก และตรวจสอบคำเตือนเตือนในคอมไพเลอร์เสมอก่อนคอมไพล์เสร็จ

---

### Phase 2: WebView & Rendering Engine Bridge (ส่วนสำคัญที่สุด)
*   **สิ่งที่ต้องทดสอบ (Test Cases):**
    - การรับส่ง handle ของ GPU shared texture ระหว่างเธรด Servo และ Slint
    - การแปลงพิกัดเคอร์เซอร์เมาส์ (Mouse Hover และพิกัด Element ใน DOM) ว่าป้อนตรงตำแหน่งปุ่มบนหน้าเว็บหรือไม่
    - การตอบสนองของ Keyboard input สำหรับแป้นเฉพาะ (เช่น Enter, Backspace, Arrow Keys)
*   **วิธีการทดสอบ (Testing Method):**
    - `Integration Test`: สร้างโมดูลเรนเดอร์ทดสอบหน้า HTML ท้องถิ่น (Local Mock HTML) ที่มีปุ่มและช่องใส่ข้อความ จากนั้นรันโค้ดจำลองคลิก (Mock Input Injection) แล้วยืนยันว่า Servo มีการส่งสัญญาณป้อนกลับของการเปลี่ยนสถานะปุ่ม
*   **ความเสี่ยงและวิธีป้องกัน (Failure Mode & Mitigation):**
    - *ความเสี่ยง:* Servo สุ่มหยุดวาดภาพเนื่องจากการเรียกใช้ OpenGL context ข้ามเธรดที่ผิดพลาด
    - *วิธีป้องกัน:* ห้ามส่ง Raw Context หรือสั่งเปลี่ยนคำสั่ง Draw ข้ามเธรด ให้ผ่าน Texture ID หรือ Shared Pointer เสมอ และควบคุมการเรียกใช้ภาพผ่านคิวงานเธรดเดียว (`Main/Render Thread`)

### Phase 3: Navigation, Address Bar & Smart Search
*   **สิ่งที่ต้องทดสอบ (Test Cases):**
    - อินพุต URL บาร์แบบแปลกๆ เช่น ลิงก์ที่ไม่มีโปรโตคอล (`google.com`), มีพอร์ตแปลกๆ (`localhost:8080`), หรือพิมพ์ภาษาไทยปะปน
    - ประสิทธิภาพ API Autocomplete เมื่อเครือข่ายทำงานช้าหรือไม่มีอินเทอร์เน็ต (เบราว์เซอร์ห้ามค้างหรือมีบั๊กแฮงก์)
*   **วิธีการทดสอบ (Testing Method):**
    - `Unit Test`: ตรวจสอบตัววิเคราะห์ URL (`UrlParser`) ด้วยชุดทดสอบ Parameterized Test (รวมค่า Input มากกว่า 50 รูปแบบ)
    - `Integration Test`: จำลองการตัดเชื่อมต่ออินเทอร์เน็ตชั่วคราวและยิง Request ไปยัง Autocomplete เพื่อดูการตอบสนองแบบ fallback
*   **ความเสี่ยงและวิธีป้องกัน (Failure Mode & Mitigation):**
    - *ความเสี่ยง:* แอปแครชเมื่อผู้ใช้พิมพ์ตัวอักษรพิเศษที่ไม่ได้ทำการแปลงรหัส (URL encoding)
    - *วิธีป้องกัน:* ใช้ไลบรารีวิเคราะห์ที่เป็นมาตรฐานของ Rust เช่น `url` crate และใช้คำสั่งห่อหุ้มข้อผิดพลาดอย่างระมัดระวัง (หลีกเลี่ยงการใช้ `.unwrap()`)

### Phase 4: Arc-Style Window & Workspace Management
*   **สิ่งที่ต้องทดสอบ (Test Cases):**
    - ความคงอยู่ของข้อมูล (State Consistency) เมื่อผู้ใช้ย้ายแท็บจาก Space A ไป Space B
    - การประมวลผลและการใช้ RAM เมื่อผู้ใช้สลับระหว่าง 50 แท็บขึ้นไป
    - ฟังก์ชันการคืนหน่วยความจำของแอปพลิเคชัน (Auto-Archive)
*   **วิธีการทดสอบ (Testing Method):**
    - `Unit Test`: เขียนจำลองขั้นตอน Tab History Stack ว่าสอดคล้องกับการเคลื่อนย้าย Space หรือไม่
    - `Manual Test`: เปิดหน้าเว็บที่กินทรัพยากรสูงหลายหน้าจอพร้อมกัน ตรวจดูว่าแท็บที่ถูก "พิน" (Pin) จะต้องไม่มีทางถูกระบบทำลายทิ้งเพื่อล้างแรม
*   **ความเสี่ยงและวิธีป้องกัน (Failure Mode & Mitigation):**
    - *ความเสี่ยง:* เกิด Stack Overflow เมื่อจัดเก็บประวัติการนำทางแท็บแบบย้อนกลับที่มีลูปวงกลม
    - *วิธีป้องกัน:* จำกัดระดับความลึกสูงสุดของประวัติย้อนหลัง (Back-Forward limits) ที่ 50 หน้า และจัดเก็บเป็น List มากกว่า Tree สปินโครงสร้าง

### Phase 5: Brave-Style Shields & Privacy Engine
*   **สิ่งที่ต้องทดสอบ (Test Cases):**
    - ความถูกต้องของการบล็อกลิงก์โฆษณา (เช่น ทรัพยากรจาก `ads.youtube.com` ต้องถูกสกัดกั้นสำเร็จ 100%)
    - เวลาประมวลผลการเช็ค URL กับ EasyList (ต้องน้อยกว่า 0.5ms ต่อ 1 Request)
*   **วิธีการทดสอบ (Testing Method):**
    - `Unit Test`: โหลดไฟล์จำลองกฎ EasyList และป้อนค่าทดสอบ URL สุ่ม 1,000 รายการ ตรวจผลลัพธ์ว่าการวิเคราะห์ตรงกับผลบล็อกที่คาดไว้
*   **ความเสี่ยงและวิธีป้องกัน (Failure Mode & Mitigation):**
    - *ความเสี่ยง:* การโหลด EasyList ทั้งหมด (ซึ่งมีกฎมากกว่า 100,000 ข้อ) เข้าสแกนสตริงทีละข้อด้วย Regex ปกติจะทำให้เบราว์เซอร์ช้าลงจนเปิดหน้าเว็บแทบไม่ได้
    - *วิธีป้องกัน:* บันทึกและดัชนีผ่านโครงสร้างข้อมูล **Bloom Filter** หรือต้นไม้แบบทรีเซตค้นหาด่วน (`Trie structure`) จาก crate `adblock` เพื่อให้ค้นพบประวัติแบบรวดเร็วและใช้แรมน้อยลง

### Phase 6: CocCoc-Style Media Downloader
*   **สิ่งที่ต้องทดสอบ (Test Cases):**
    - การจับคู่ชนิดไฟล์ (Mime-Type matching) จาก Response Header
    - การประกอบไฟล์พาร์ทต่างๆ หลังจากดาวน์โหลดเสร็จสิ้น (เช็คว่าไม่มีข้อมูลส่วนใดเสียหาย)
    - ฟังก์ชันการกู้คืนการดาวน์โหลดต่อ (Resume) จากเซิร์ฟเวอร์ที่รองรับและไม่รองรับ Range
*   **วิธีการทดสอบ (Testing Method):**
    - `Unit Test`: จำลองเซิร์ฟเวอร์ทดสอบการส่งช่วงไฟล์ทีละไบต์ (Mock Range Server) เพื่อทดสอบ Downloader Client ใน Rust
*   **ความเสี่ยงและวิธีป้องกัน (Failure Mode & Mitigation):**
    - *ความเสี่ยง:* เมื่อเครือข่ายล่มกลางคัน เธรดย่อยทั้งหมดปิดตัว แต่ไฟล์พาร์ทที่ดาวน์โหลดค้างไว้ถูกลบทำลายหมด
    - *วิธีป้องกัน:* บันทึกประวัติและพาร์ทที่โหลดไปแล้วลงในไฟล์ชั่วคราวที่มีนามสกุลแยก (เช่น `.part`) พร้อมเขียนข้อมูลการเชื่อมต่อลง SQLite เสมอเพื่อให้เปิดแอปมารันต่อได้

### Phase 7: Profile & Session Isolation
*   **สิ่งที่ต้องทดสอบ (Test Cases):**
    - คุกกี้จากโปรไฟล์ส่วนตัว (Personal Profile) จะต้องไม่มีวันหลุดไปปรากฏในโปรไฟล์ทำงาน (Work Profile)
    - โหมดไม่ระบุตัวตน (Incognito) ต้องไม่สร้างโฟลเดอร์หรือบันทึกไฟล์ชั่วคราวใดๆ บน Disk
*   **วิธีการทดสอบ (Testing Method):**
    - `Integration Test`: ล็อกอินบัญชีทดสอบในโปรไฟล์ A จากนั้นรันโค้ดดึงคุกกี้ในโปรไฟล์ B ยืนยันว่าผลลัพธ์เป็นค่าว่าง
*   **ความเสี่ยงและวิธีป้องกัน (Failure Mode & Mitigation):**
    - *ความเสี่ยง:* แคชระบบของไลบรารีเครื่องยนต์เน็ตเวิร์กยังเก็บหน่วยความจำร่วมกันที่ระดับ Global Singleton
    - *วิธีป้องกัน:* ตั้งค่าอินสแตนซ์ HTTP Client หรือ Servo Network Context แยกกันโดยผูก path ของแคชไปที่โฟลเดอร์แยกต่างหากอย่างชัดเจนเมื่อสร้าง Profile Instance

### Phase 8: Web Developer Tools
*   **สิ่งที่ต้องทดสอบ (Test Cases):**
    - การดึงข้อมูล DOM ที่มีขนาดใหญ่มาก (มากกว่า 10,000 Node) ต้องไม่ทำให้หน้าจอค้าง
    - ข้อมูล Console log ห้ามตกหล่นหรือมีการส่งวนลูปวนซ้ำ (Circular reference log loop)
*   **วิธีการทดสอบ (Testing Method):**
    - `Unit Test`: จำลองข้อมูล DOM รูปแบบมีลูปและทดสอบระบบ Serialization ว่าไม่เกิดการระเบิดของหน่วยความจำ
*   **ความเสี่ยงและวิธีป้องกัน (Failure Mode & Mitigation):**
    - *ความเสี่ยง:* การส่ง DOM Tree ขนาดใหญ่ผ่านช่องทางสื่อสาร IPC UI ทำให้ UI ค้างเป็นวินาที
    - *วิธีป้องกัน:* ส่งเฉพาะ Node ที่ปรากฏบน viewport ปัจจุบัน (Lazy loading DOM Tree) และแปลงเป็นโครงสร้างแบบแบนราบที่มีความลึกจำกัด

### Phase 9: Web Extension System (Manifest V3)
*   **สิ่งที่ต้องทดสอบ (Test Cases):**
    - ตัวแทรก Script (Script Injector) จะต้องไม่สามารถหนีหลุดออกจากกรอบความปลอดภัย (Sandbox Escape) เข้ามาเข้าถึงระบบไฟล์หลักของเครื่องผู้ใช้ได้
*   **วิธีการทดสอบ (Testing Method):**
    - `Integration Test`: เขียน Extension จำลองพยายามเข้าถึง API ของระบบปฏิบัติการผ่านช่องทาง JavaScript แล้วเช็คว่าถูกปฏิเสธกลับมาทั้งหมด
*   **ความเสี่ยงและวิธีป้องกัน (Failure Mode & Mitigation):**
    - *ความเสี่ยง:* สิทธิ์การเข้าใช้งาน APIs ของบราวเซอร์รั่วไหล
    - *วิธีป้องกัน:* จำกัดขอบเขตสิทธิ์ของ JavaScript Sandbox โดยสมบูรณ์ผ่านระดับความปลอดภัยของ Servo Spidermonkey API

### Phase 10: Multi-Process & Sandboxing
*   **สิ่งที่ต้องทดสอบ (Test Cases):**
    - เมื่อกระบวนการเรนเดอร์ (Renderer Process) แครช บราวเซอร์หลักจะต้องไม่ดับตามไปด้วย
*   **วิธีการทดสอบ (Testing Method):**
    - `Manual Test`: รันคำสั่ง kill process ของเรนเดอร์ย่อย ยืนยันว่าหน้าแท็บหลักแสดงข้อผิดพลาดแต่วินโดว์บราวเซอร์ยังสามารถปิด เปิดแท็บใหม่ หรือใช้งานส่วนอื่นได้ปกติ
*   **ความเสี่ยงและวิธีป้องกัน (Failure Mode & Mitigation):**
    - *ความเสี่ยง:* ความล่าช้าในการส่งข้อมูลผ่านท่อ IPC (IPC overhead) ส่งผลให้การพิมพ์ตอบสนองช้า
    - *วิธีป้องกัน:* ออกแบบแพ็คเกจข้อมูล IPC ให้มีขนาดกะทัดรัด (Compact binary serialization เช่น `bincode` หรือ `protobuf`)

### Phase 11: OS Integration & Performance
*   **สิ่งที่ต้องทดสอบ (Test Cases):**
    - การสลับเอาต์พุตเสียง (Audio device switching) ทันทีขณะเล่นภาพยนตร์
*   **วิธีการทดสอบ (Testing Method):**
    - `Manual Test`: เสียบและถอดหูฟังระหว่างการทดสอบเล่นวิดีโอ เพื่อดูการฟื้นฟูเธรดเสียง
*   **ความเสี่ยงและวิธีป้องกัน (Failure Mode & Mitigation):**
    - *ความเสี่ยง:* แหล่งระบบถอดรหัสวิดีโอเกิดการระเบิดหน่วยความจำรั่วซึม (Codec Memory Leak)
    - *วิธีป้องกัน:* ติดตามทรัพยากรผ่านเทคนิค **RAII (Resource Acquisition Is Initialization)** ของ Rust ครอบคลุมไลบรารีภายนอกเสมอ

### Phase 12: Packaging, Tests & Delivery
*   **สิ่งที่ต้องทดสอบ (Test Cases):**
    - ลายเซ็นดิจิทัล (Digital Signature/Signing) ของโปรแกรมติตตั้งบราวเซอร์บน Windows/macOS ต้องไม่มีข้อความแจ้งเตือนจากระบบรักษาความปลอดภัยว่าโปรแกรมนี้เป็นอันตราย (SmartScreen หรือ Gatekeeper)
*   **วิธีการทดสอบ (Testing Method):**
    - `CI/CD pipeline test`: รันสคริปต์การทำ package บิลด์ในสภาพแวดล้อมที่จำลอง และทดสอบติดตั้งบนระบบปฏิบัติการที่สดใหม่
*   **ความเสี่ยงและวิธีป้องกัน (Failure Mode & Mitigation):**
    - *ความเสี่ยง:* บิลด์ไม่เสถียรใน OS บางเวอร์ชันเนื่องจากคอมไพล์ไลบรารีระบบด้วย static-link ที่ไม่สมบูรณ์
    - *วิธีป้องกัน:* ใช้เทคนิค Docker container บิลด์หรือ macOS Virtual Machines ที่มีระบบ OS สะอาด เพื่อยืนยันว่าไม่มีการผูกไลบรารีภายในเครื่องนักพัฒนาคนนั้นๆ ออกไปด้วย
